# Подробный анализ противоречий в описании процесса

## Противоречие 1: Закрытие заявки в Системе №2

### Описание проблемы

В исходном описании сказано:
> "Техническая поддержка выдаёт это оборудование пользователю и закрывает заявку пользователя в системе №1 и закрывает заявку в системе №2."

Это утверждение применимо к обоим сценариям (с наличием и отсутствием оборудования). Однако в сценарии, когда оборудование заказывается у поставщика, процесс значительно более сложный.

### Вопросы

1. **Когда именно закрывается заявка в системе №2?**
   - Сразу после отправки заказа поставщику?
   - После получения оборудования от поставщика?
   - После передачи оборудования логистике?
   - После получения пользователем?

2. **Кто инициирует закрытие?**
   - Отдел снабжения?
   - Техподдержка?
   - Система автоматически?

3. **Есть ли промежуточные статусы?**
   - "В ожидании поставки"
   - "Получено от поставщика"
   - "Отправлено логистике"

### Рекомендация

Необходимо определить четкую последовательность:
```
Заказ → Ожидание → Получение от поставщика → Отправка логистике → Получение пользователем → Закрытие
```

И указать, на каком этапе обновляется статус в системе №2.

## Противоречие 2: Роль и взаимодействие логистики

### Описание проблемы

Логистика упоминается дважды:
- "снабжение вызывает логистику и отправляет оборудование первой линии технической поддержки"
- "вызывает логистику и передаёт оборудование технической поддержке"

Но остается множество неясностей.

### Вопросы

1. **Как логистика получает информацию о вызове?**
   - Через систему (SMS, email, push-уведомление)?
   - По телефону?
   - По электронной почте?
   - Через какую-то систему управления логистикой?

2. **Какие данные передаются логистике?**
   - Адрес доставки?
   - Описание оборудования?
   - Номер заявки?
   - Приоритет доставки?

3. **Фиксирует ли логистика действия в системах?**
   - Подтверждает ли получение оборудования?
   - Обновляет ли статус в системе №2?
   - Отправляет ли уведомление пользователю?

4. **Какое время доставки?**
   - В тот же день?
   - На следующий день?
   - Зависит от адреса?

### Рекомендация

Необходимо документировать:
- Процесс вызова логистики (API, система, или ручной процесс)
- Данные, которые передаются
- Статусы доставки, которые логистика может сообщать
- Время доставки в зависимости от адреса

## Противоречие 3: Подтверждение получения пользователем

### Описание проблемы

Описание не упоминает явный этап подтверждения получения оборудования пользователем. Сказано только:
> "Техническая поддержка выдаёт это оборудование пользователю и закрывает заявку"

### Вопросы

1. **Как L1 узнает, что пользователь получил оборудование?**
   - Пользователь звонит/пишет в L1?
   - L1 звонит пользователю?
   - Пользователь подтверждает в системе №1?

2. **Что происходит, если пользователь не получил оборудование?**
   - Заявка остается открытой?
   - Создается новая заявка?
   - Логистика должна вернуть оборудование?

3. **Есть ли проверка работоспособности оборудования?**
   - Пользователь проверяет на месте?
   - Есть ли гарантия?
   - Кто отвечает за неработающее оборудование?

### Риск

Если заявка закрывается без подтверждения пользователя, возможны ситуации:
- Пользователь не получил оборудование, но заявка закрыта
- Оборудование неработающее, но заявка закрыта
- Нет возможности отследить проблему

### Рекомендация

Добавить явный этап подтверждения:
```
L1 выдает оборудование → Пользователь подтверждает получение (в системе или по телефону) → L1 закрывает заявку
```

## Противоречие 4: Сценарии сбоя поставки

### Описание проблемы

Описание не предусматривает сценарии, когда что-то идет не по плану:
- Поставщик не может поставить оборудование
- Оборудование повреждено при доставке
- Поставщик отказывает в поставке
- Оборудование не соответствует спецификации

### Вопросы

1. **Что происходит, если поставщик не может поставить оборудование?**
   - Кто уведомляет пользователя?
   - Ищется ли альтернативный поставщик?
   - Заявка закрывается или остается открытой?

2. **Кто отвечает за повреждение при доставке?**
   - Логистика?
   - Поставщик?
   - Кто возмещает убыток?

3. **Есть ли SLA для поставки?**
   - Максимальное время ожидания?
   - Что происходит, если превышено?

### Рекомендация

Добавить обработчики исключений:
```
Заказ → Поставщик не может поставить → Уведомить пользователя → Поиск альтернативы ИЛИ закрытие заявки
```

## Противоречие 5: Механизм вызова логистики

### Описание проблемы

Фраза "снабжение вызывает логистику" не конкретизирует, как это происходит.

### Вопросы

1. **Это интеграция между системами?**
   - Система №2 автоматически отправляет заказ в систему логистики?
   - Или это ручной процесс?

2. **Есть ли система управления логистикой?**
   - Если да, то как она интегрирована с системой №2?
   - Если нет, то как отслеживается статус доставки?

3. **Есть ли подтверждение получения заказа логистикой?**
   - Логистика подтверждает, что получила заказ?
   - Система №2 обновляется?

### Рекомендация

Необходимо документировать:
- Тип интеграции (API, система, ручной процесс)
- Время отклика логистики
- Способ отслеживания доставки

| Закрытие заявки в системе №2 | Высокая | Неясность в статусе заявки |

| Сценарии сбоя | Средняя | Отсутствие плана действий |
